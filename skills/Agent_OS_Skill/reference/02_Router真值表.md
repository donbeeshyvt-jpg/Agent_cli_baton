# 02 ｜ Router 真值表與旗標契約（把「選哪個角色」變成查表）

> 紅隊指出第一版 router 是散文，小模型靠心算會跳關。這份把它變成**確定性的查表規則**：意圖解碼者吐一組固定旗標 → 查表點亮角色 → 照不變式排序。
>
> 這是讓 router「**可機械執行**」的唯一辦法，也直接補上「12 角色裡沒人是 router 本身」的缺口——router 不是一個角色，而是這張表，由主控引擎執行。

---

## A. 旗標契約（意圖解碼者的固定輸出 schema）

意圖解碼者在「問題說明書」尾端**必須**附上這段機器可讀欄位，下游 router 才有東西可查：

```json
{
  "真實目標": "<一句話>",
  "成功標準": "<可驗證的驗收條件>",
  "答錯成本": "高 | 中 | 低",
  "可逆性": "可逆 | 不可逆",
  "flags": {
    "NEEDS_PLAN":     false,   // 多個互相依賴的部分 / 大而模糊 / 需里程碑
    "NEEDS_SCHEDULE": false,   // 要時程、提醒、里程碑（時間到了由【人】動手）
    "NEEDS_LOOP":     false,   // 定期、自己跑下去、持續監控、無人值守（時間到了由【迴圈自己】執行+驗證+推進）；三特徵至少中兩項才掛：會重複發生/單次做完沒意義/要跨session持續。灰帶預設只掛 NEEDS_SCHEDULE
    "NEEDS_EVIDENCE": false,   // 答案依賴外部事實 / 官方規範 / 版本時效 / 找 repo / 論文
    "NEEDS_BUILD":    false,   // 要產出方案 / 寫程式 / 改檔 / 落地
    "NEEDS_DEBUG":    false,   // 有 bug / 報錯 / 生產事故 / 行為不符預期
    "HIGH_STAKES":    false    // 答錯成本高 / 不可逆 / 引用可查證事實 / 要對外交付
  }
}
```

> 小模型友善寫法：直接在說明書最後寫「旗標：NEEDS_BUILD=true, NEEDS_DEBUG=true, HIGH_STAKES=true，其餘 false」即可。

## B. 旗標 → 點亮角色 真值表

| 旗標 | 點亮的角色 | 備註 |
| --- | --- | --- |
| （永遠） | ① 意圖解碼者 | 除非「封閉單點事實 + 答錯成本低 + 無未定義名詞」可跳過 |
| `NEEDS_PLAN` | ② 任務架構師 | 大/多依賴才需要；單線任務免 |
| `NEEDS_SCHEDULE` | ③ 規劃排程者 | 需 cron/執行框架；無排程後端則退化為靜態時間清單。分界判別句：「時間到了，是【人】動手（SCHEDULE）還是【迴圈自己】執行+驗證+推進（LOOP）？」 |
| `NEEDS_LOOP` | 載入 `Loop_Engineering_Skill`（迴圈工程規劃管線，12 角色照它的接力地圖跑） | 產出「迴圈工程規劃書」而非一次性答案；**硬規則：一律 L1 report-only 出生，升級走 APPROVED 批准**；灰帶預設只掛 NEEDS_SCHEDULE。HIGH_STAKES 加判：候選 pattern 的 risk≥medium／token_cost≥high／含 L2/L3 推進目標，任一命中即 HIGH_STAKES=true（⑧⑩不可省；聯集正典見 Loop_Engineering_Skill/reference/02_角色映射與觸發.md）。**若本環境讀不到 Loop_Engineering_Skill → 降掛 NEEDS_SCHEDULE 並明告使用者缺件，絕不憑記憶模擬該管線** |
| `NEEDS_EVIDENCE` | ④ 研究總管 ⇉ ⑤ 領域透鏡研究員 ×N | 單一面向小考據可只用一個透鏡、不驚動總管 |
| `NEEDS_BUILD` | ⑥ 方案提案者 →（選定）→ ⑦ 最小變更實作者 | 路徑唯一無爭議時可略過提案者 |
| `NEEDS_DEBUG` | ⑨ 系統化除錯者 | **硬規則：沒重現根因前不准修** |
| `HIGH_STAKES` | ⑧ 紅隊驗證者 ＋ ⑩ 驗收實證者 | **硬規則：HIGH_STAKES 亮起，這兩個不可省** |
| （收尾，幾乎永遠） | ⑪ 綜合編織者 → ⑫ 記憶技能策展人 | 純機器中繼 / 一次性瑣碎可略 |

## C. 排序不變式（骨幹）

```
① 意圖解碼者
 → ② 任務架構師
 →（③ 規劃排程者，若 NEEDS_SCHEDULE）
 →〔④ 研究總管 ⇄ (⑤ 領域透鏡 ×N)〕
 → ⑥ 方案提案者 →（選定）→ ⑦ 最小變更實作者
 →〔⑨ 系統化除錯者，若 NEEDS_DEBUG〕
 → ⑧ 紅隊驗證者 ⇄ 回補
 →〔⑩ 驗收實證者 gate，若 HIGH_STAKES〕
 → ⑪ 綜合編織者
 → ⑫ 記憶技能策展人
```

**Tie-break（同時亮多旗標）**：永遠先消「最便宜、最上游」的不確定性 →
`定義不確定性(①) → 結構不確定性(②) → 事實不確定性(④) → 建構/落地(⑦) → 對抗式驗證(⑧⑩)`

## D. 可裁剪捷徑（小模型省 context 用）

| 情境 | 精簡路徑 |
| --- | --- |
| 封閉單點事實（「現在 UTC 幾點」）| ①(輕量) → 工具 → ⑪(一句話) |
| 純研究無建構 | ① → ④⇄⑤ → ⑧ → ⑪（砍 ⑥⑦⑨）|
| 純落地小改且路徑明確 | ① → ⑦ → ⑩（砍 ②④⑥⑧）|
| 純抓錯 | ① → ⑨ → ⑩ → ⑪ |

## E. 迴圈退出條件（防小模型死循環 / 假完成）

- **研究 ⇄ 紅隊迴圈**：滿足任一即放行 →（a）所有 `HIGH_STAKES` 主張信心達「高」；（b）迴圈滿 **2 輪**仍無新證據。
- **單一主張飽和**：獲 **≥2 個獨立一手來源**即停查。
- **每角色工具呼叫上限**：建議 leaf 8–12 次、orchestrator 偏低（只 routing）；逼近上限仍未收斂 → 不硬撐，輸出「已達成／未達成／卡點」三段式中繼交回上游。
- **`<tool_persistence>` 細則**：工具回空要**換策略重試**，不是同策略重撞。
