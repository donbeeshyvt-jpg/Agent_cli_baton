# 迴圈模式庫｜12 個可規劃的迴圈模板（patterns/迴圈模式庫.md）

> **這是模式庫，不是教科書。** 用法：先走「選型決策樹」選出 1–2 個模式 → 複製該模式的規格單 → 照 📋 fill-in 欄位填完 → 交棒給③規劃排程者排節奏 → 每支迴圈上線前走一張 **APPROVED-MMM**（MMM 取 `docs/REQUEST_LOG.md` 下一個流水號，與 LOOP 編號無關；填寫前先開 REQUEST_LOG 確認該列存在）。
> 世界觀雙引號並陳：「別再逐回合 prompt agent；設計會自己 prompt agent 的迴圈，你的工作是寫迴圈」——但「無人看管的迴圈，也會無人看管地犯錯…蓋迴圈，要像打算留在工程師位子上的人，而不是只按 go 的人」。**迴圈是控制系統，不是自動駕駛。**

---

## 〇、管線定位（本庫在哪一步被打開）

迴圈工程規劃管線 = ①意圖解碼(分診+掛旗) → ②任務架構(工程MECE成迴圈候選+依賴) → ③規劃排程(第一次上場：節奏/Tier草案) → ④研究總管⇉⑤透鏡(此工程迴圈化的證據+反例) → ⑥方案提案(每候選的模式選型+取捨表) → ③(第二次上場：艦隊預算加總與錯峰) → ⑦最小變更實作(起草每迴圈的執行 prompt 草稿，含 denylist 一句式) → ⑨系統化除錯(每迴圈的故障除錯協定+回滾路徑，如 git revert 級) → ⑧紅隊(攻擊設計：token債/碰撞/驗證缺口/假完成) → ⑩驗收(Loop Ready 檢核，預設 NEEDS WORK) → ⑪綜合編織(迴圈工程規劃書) → ⑫策展(沉澱可複用模板)。

- **③規劃排程者在管線中上場兩次：第一次在②任務架構之後出節奏/Tier 草案；第二次在⑥方案提案之後做艦隊預算加總與錯峰。**
- 本庫主要服務 ②任務架構（以「一 cadence 一主迴圈帶、錯峰才准並跑」為拆分邊界）與 ⑥方案提案（不同模式 × 自治級起點構成取捨表）；填完的規格單就是 LOOP-NNN.md 初稿，交給③第二次上場做艦隊預算加總與錯峰。
- 候選 pattern 是否 **HIGH_STAKES**：判準全文只寫一次，見 `reference/02_角色映射與觸發.md`——命中者 ⑧紅隊與⑩驗收不可省。

## 〇之二、治理文件三件套與每輪固定六步（先讀這節再選型）

### 治理文件三件套（docs/loops/ 下只有這三類檔）

| 檔案 | 內容 | 紀律 |
| --- | --- | --- |
| `docs/loops/LOOP-NNN.md` | 單迴圈定義（目的、約束、denylist、**預算內嵌 §8**——§8 必含「每輪子代理數上限」「子代理鏈深度上限」兩列） | frontmatter 批准欄 `approved_by: APPROVED-{{MMM}}`（MMM 取 `docs/REQUEST_LOG.md` 下一個流水號，與 LOOP 編號無關；填寫前先開 REQUEST_LOG 確認該列存在） |
| `docs/loops/STATE-NNN.md` | 持久狀態，**與 LOOP-NNN 同號**；Run Log 附於其尾節「## Run Log」，**每輪一行 JSON**；frontmatter 含 `last_digest_ack:{日期/owner}`（人類讀完 digest 手動回填——此處需人工補做） | 每 pattern 一檔，絕不多迴圈共寫；**禁止 `*-state.md` 命名** |
| `docs/loops/LOOP_PORTFOLIO.md` | 艦隊總表＋艦隊級約束節（denylist 基線、`loop-pause-all` 旗標、衝突優先序、錯峰規則）＋艦隊健康度＋跨迴圈 Human Inbox | 艦隊級狀態一律住這裡 |

**不存在**獨立的 `loop-budget.md`／`loop-run-log.md`／`loop-constraints.md`——預算住 LOOP-NNN §8 與 PORTFOLIO 總表；Run Log 住 STATE-NNN 尾節；艦隊約束住 PORTFOLIO。三件套模板全文見 `reference/01_迴圈文件契約.md`。

### 每輪固定六步（跳過任一步＝該輪無效）

1. **讀 `docs/loops/LOOP_PORTFOLIO.md`**：查 `loop-pause-all` 與自身 status，非 active 立即退出。同步檢查理解債：本迴圈 STATE-NNN.md frontmatter 的 `last_digest_ack` 落後 Last run **≥3 輪** → 本輪自動降為 no-op，並在 digest 頂標紅「**digest 連續未讀，迴圈已自動降速**」。
2. **讀 `docs/loops/LOOP-NNN.md`** 的約束與預算（§8）。
3. **讀 `docs/loops/STATE-NNN.md`**（本迴圈持久狀態）。
4. **掃其他 `STATE-*.md` 的 `acting_on`**——衝突 → 本輪讓位並記 Run Log。
5. **行動前寫 STATE**（宣告 `acting_on`）。
6. **收尾寫 Run Log**（STATE-NNN.md 尾節「## Run Log」一行 JSON）＋**人類必讀 digest**。

## 🚨 使用本庫的強制流程（NEVER SKIP）

1. **先問 Q0（迴圈 vs 一次性）再選模式**——Loops discover ongoing work; goals finish bounded tasks。一次性任務包成迴圈＝直接打回。
2. **每支迴圈必須寫下「為什麼不用一次性任務就好」**——這句話進 LOOP-NNN.md 的目的欄，對治 intent debt（迴圈理由未記錄）。
3. **任何新迴圈一律 L1 report-only 出生**——不管規格單上「安全級起點」寫多高，那是「畢業起點」不是「出生點」。Never skip L1 for a new pattern on a production repo。
4. **同節奏帶允許多支迴圈的唯一條件＝錯峰啟動（相位差≥半個 cadence）＋各自宣告 acting_on＋CI 紅色期間 PR Babysitter 自動 no-op 讓位 CI Sweeper**；每 pattern 一個 `docs/loops/STATE-NNN.md`（與 LOOP-NNN 同號），絕不多迴圈共寫一檔。
5. **禁止迴圈級聯（鐵律）：任何迴圈不得觸發另一支迴圈；迴圈間只透過 STATE/PORTFOLIO 文件與人類閘門互動。**
6. **maker 與 checker 必須是不同 context 的角色／子代理**（若執行框架支援生子代理→分派給獨立子代理；否則在單一模型內切換角色並清空上一角色立場）——worker does not grade its own homework。
7. **規格單一欄都不准留白**——填不出來的欄位明標「此處需人工補做」，絕不假裝。
8. **選完模式必須走一遍結尾的「多迴圈組合守則」**——即使只選了一支（因為未來會擴編，碰撞規則現在就要寫進 LOOP-NNN.md）。

## 一、模式選型決策樹（任務重複性 × 風險 × 成本）

### Q0：這根本該不該是迴圈？（一票否決）

```
Q0. 這是「有明確邊界、做完就結束」的任務嗎？
 ├─ 是 → 🚫 停。不要建迴圈。回到 Agent OS 一次性流程（NEEDS_PLAN/NEEDS_BUILD…）。
 │        判別句一：做完這一次之後，同樣的工作還會不會自己長出來？不會 → 不是迴圈。
 │        判別句二（LOOP vs SCHEDULE 分界）：時間到了，是【人】動手（SCHEDULE）
 │        還是【迴圈自己】執行+驗證+推進（LOOP）？人動手 → 只掛 NEEDS_SCHEDULE。
 └─ 否（工作會反覆發生／需要持續監看／「沒人開終端時也該發生」）→ 進 Q1。
```

> 上游掛旗紀律：語彙訊號只是線索，NEEDS_LOOP 的三特徵判準（(a)任務會重複發生 (b)單次做完沒意義 (c)要跨 session 持續+驗證+推進——**至少命中兩項**才掛旗）、反向出口必答與灰帶處理，全文見 `SKILL.md` 觸發段與 `reference/02_角色映射與觸發.md`。本庫假設進場者已通過該判準。

### Q1：現在最痛的是什麼？（What hurts right now? → 模式對映）

```
Q1. 現在最痛的是什麼？
 ├─ 每天早上開工一團亂、訊號散落各處沒人彙整 ──→ LP-01 每日分診 Daily Triage ★不確定時的預設答案
 ├─ PR 積壓、review 卡住、CI 失敗沒人理 ────────→ LP-02 PR 保姆 PR Babysitter
 ├─ main 的 CI 常紅、大家對紅 main 麻木 ────────→ LP-03 CI 清道夫 CI Sweeper（⚠ 有前置條件，見規格單）
 ├─ 依賴過期、Dependabot／CVE 噪音淹沒 ─────────→ LP-04 依賴清掃 Dependency Sweeper
 ├─ release notes／changelog 老是過期 ──────────→ LP-05 變更日誌起草 Changelog Drafter（最便宜起手式之一）
 ├─ merge 債、TODO、dead code 堆積 ─────────────→ LP-06 合併後清理 Post-Merge Cleanup
 ├─ issue 噪音大、重複票、標籤混亂 ─────────────→ LP-07 議題分診 Issue Triage
 ├─ 「每週都得追 X 主題的新進展」───────────────→ LP-08 研究監測 Research Watch
 ├─ 資料常態流入、品質默默飄移 ─────────────────→ LP-09 資料品質 Data Quality Sweeper
 ├─ 定期要產固定格式的內容（週報／文件章節）────→ LP-10 內容產線 Content Pipeline
 ├─ docs/ 紀錄（TASKS／CODE_INDEX／HANDOFF／
 │   LOOP↔STATE）與專案現實漂移 ────────────────→ LP-11 專案文件同步 Docs Sync
 └─ 同樣的事已經教過 agent 兩次以上 ────────────→ LP-12 記憶沉澱 Memory Curation
```

### Q2 / Q3：用風險軸與成本軸修正選型

| 軸 | 檢查問題 | 答「高」時的規劃動作 | 答「低」時的規劃動作 |
| --- | --- | --- | --- |
| **重複性** | 這個痛多久發生一次？ | 每小時／每日發生 → 短節奏模式候選（LP-02/03/04） | 週／月級 → 只選 1d 以上節奏模式（LP-01/05/08/11/12），或乾脆不建迴圈 |
| **風險** | 迴圈會碰到 production／金流／auth／不可逆操作嗎？答錯成本高嗎？ | 起點鎖死 L1、人類閘門全開、denylist 先寫好（`.env*`、`**/secrets/**`、`auth/**`、`payments/**`、`**/migrations/**`…）、每次升格走 APPROVED-MMM | 可以規劃 L1→L2 的畢業路徑，但仍要 verifier 與 attempt cap |
| **成本** | token 預算緊嗎？（實證基線：multi-agent ≈ 15× 單次 chat；50 步 >30×；模型自估花費不可信 corr≤0.39） | 避開 5–15m 高頻模式（LP-02/03），選 1d 低頻（LP-01/05/06/08/12）；全部 early_exit 必須 | 仍然要設日上限與斷路器——2026 年的失敗特徵是 fail expensively rather than loudly（$437 過夜、$6,531 AWS 帳單都有實案） |

風險或成本任一答「高」→ 對照 `reference/02_角色映射與觸發.md` 的 HIGH_STAKES 判準（全文只住那裡）；命中即 HIGH_STAKES=true，⑧紅隊與⑩驗收不可省。

**不確定就選 LP-01 每日分診 @ L1**——原典原話：「If unsure, start with Daily Triage at L1. It teaches state discipline without auto-merge risk.」**非 coding 專案的預設首發＝LP-08 或 LP-11**（兩者天生 report-only）。

## 二、規格單欄位 schema（逐欄填寫指引）

| 欄位 | 填什麼 | 填寫紀律 |
| --- | --- | --- |
| **名稱** | 模式名＋這個專案的實例名 | 一個實例一個名，之後就是 LOOP-NNN.md 文件的標題 |
| **目的** | 單一目標一句話 ＋ explicit non-goals ＋「為什麼不用一次性任務」 | 一句話塞不下＝範圍太大，拆掉（對治 intent debt） |
| **觸發/節奏** | cadence ＋ 降級寫法 | 標準句式：「若執行框架支援排程器→每 X；否則列入**手動重啟清單**（何時由人手動跑一輪），此處需人工補做」 |
| **輸入(讀什麼STATE)** | 每輪固定六步①〜④的讀取序列＋本模式專屬訊號源 | 一律照六步：`LOOP_PORTFOLIO.md`（pause-all／自身 status／理解債檢查）→ `LOOP-NNN.md`（約束與預算 §8）→ 本迴圈 `docs/loops/STATE-NNN.md`（與 LOOP-NNN 同號）→ 掃其他 `STATE-*.md` 的 `acting_on`；跳過任一步＝該輪無效 |
| **執行(maker做什麼)** | phases 編號步驟 | one item per loop、smallest possible diff、Never refactor unrelated code；行動前先寫 STATE 宣告 `acting_on`（六步⑤）；生子代理的條件：僅當 state 說 actionable |
| **驗證(checker怎麼擋)** | checker 的證據標準 | 預設 REJECT；驗證優先序：規則式 deterministic > 實跑／E2E（can the agent run the thing?）> LLM-as-judge（官方認證 generally not robust，限 3–5 次迭代）；**transcript 自述字串不是證據** |
| **人類閘門(何時升級人類)** | escalation triggers 清單 | risky / ambiguous → Escalate to human with full context；升級項寫進本迴圈 STATE-NNN.md 的 High Priority（含 `Loop action:` 與 `Human decision:` 欄）；跨迴圈歧義項放 `LOOP_PORTFOLIO.md` 的 Human Inbox |
| **預算上限與斷路器** | tokens_noop / report / action ＋日上限＋斷路器 | **單輪 action 預算不得超過日上限的 60%，否則該輪開始前即降為 report-only（絕不在動手中途切斷）**；斷路器三件套至少：預算 80% 轉 report-only、max 3 attempts 升級、`loop-pause-all` 旗標（住 LOOP_PORTFOLIO.md）即刻退出；early_exit（無事可做 <5k tokens 退場）標明必須與否；無計量能力時降級為結構性配額（見 6.4） |
| **每輪子代理數上限** | 一個數字 | **L1 期＝0，畢業後建議 ≤3**；此列必須同時出現在 LOOP-NNN.md §8 預算表 |
| **子代理鏈深度上限** | 固定＝1 | **子代理不得再生子代理；需更深分解→升級人類**；此列必須同時出現在 LOOP-NNN.md §8 預算表 |
| **安全級(起點)** | L1/L2/L3 的**畢業起點** | 出生一律 L1；L2 畢業需 1–2 週穩定 triage 實證；L3 需 2–4 週 L2 零意外變更＋verifier＋state＋預算三件套齊備；每次升格一張 APPROVED-MMM；升格證據對應 Loop Ready 檢核（**門檻 L1≥38/L2≥58/L3≥78，逐項權重見 reference/02_角色映射與觸發.md**），`last_digest_ack` 回填紀錄是 L1→L2 升格證據之一 |
| **退出/完成條件** | 每 run 早退條件＋整支迴圈退役條件 | 退役三步：刪排程（或從手動清單移除）→ STATE-NNN.md 標 `status: retired` 歸檔 → post-mortem 寫進 Run Log 與紀錄系統 |
| **適用訊號** | 什麼徵兆說明該用這支 | 供意圖解碼者的 NEEDS_LOOP 判斷引用（三特徵判準見 SKILL.md／reference/02） |
| **不適用訊號** | 什麼徵兆說明不該用／該換支 | 反向出口，防止為建而建 |

## 四、Part A：軟體工程七大模式（改寫自正典 registry）

### LP-01 每日分診 Daily Triage ★預設首選

| 欄位 | 內容 |
| --- | --- |
| 名稱 | LP-01 每日分診 Daily Triage |
| 目的 | 單一目標：每天一次掃描專案全部訊號源，產出「今天值得人類知道的事」四節結構化報告。**report-only；它「建議」其他迴圈的當日執行優先序，寫入 digest 與手動重啟清單，由人類決定啟動**——絕不直接觸發任何迴圈（禁止級聯鐵律）。non-goals：不做架構決策、不做 multi-file refactor、triage 只給訊號不發明工作（Never propose architectural overhauls during triage）。為什麼是迴圈：訊號每天重新長出來，一次性彙整隔天就過期。 |
| 觸發/節奏 | 若執行框架支援排程器→每日 1 次（清晨 off-peak）；否則手動重啟清單：**每日開工第一件事人工跑一輪**（此處需人工補做排程） |
| 輸入(讀什麼STATE) | 六步①〜④：`docs/loops/LOOP_PORTFOLIO.md`（pause-all／status／理解債檢查）→ `LOOP-NNN.md`（§8）→ 本迴圈 `docs/loops/STATE-NNN.md`（Last run／High Priority／Watch List／Recent Noise）→ 掃其他 `STATE-*.md` 的 `acting_on`；另讀：訊號源（issue tracker、CI 狀態、待辦、收件匣——依能力探測結果，讀不到的源明標缺口） |
| 執行(maker做什麼) | phases：1. report——一次便宜的 triage pass 掃訊號，輸出四節：High-Priority（一行描述＋why it matters today＋Suggested next action＋effort 估計）／Watch Items／Noise-Ignore／State Updates；規則：Be brutally concise、When in doubt, put it in Watch or Noise 2. 建議其他迴圈當日優先序（只寫入 digest 與手動重啟清單，人類決定啟動）3. act-small-wins（L2 畢業後才啟用：單檔、無行為變更的小勝）4. escalate |
| 驗證(checker怎麼擋) | L1 無寫入動作→checker 查報告結構四節齊全、每條 High-Priority 都答得出「為什麼今天重要」，答不出就降到 Watch；L2 的 small win 必須由 verifier 子代理在隔離環境實跑測試附輸出 |
| 人類閘門 | design decisions、multi-file refactors 一律升級；報告本身就是人類 digest——**人沒讀，迴圈就不算完成這一輪**：人類讀完後手動回填 STATE-NNN.md frontmatter 的 `last_digest_ack`（此處需人工補做）；ack 落後 ≥3 輪→迴圈自動降為 no-op（理解債防線）；⑩驗收把 ack 回填紀錄納入 L1→L2 升格證據 |
| 預算上限與斷路器 | noop 5k / report 50k / action 200k；**L1 期日上限 100k；升 L2 時日上限隨 APPROVED-MMM 重批（建議 ≥250k）**；單輪 action ≤ 日上限 60%；斷路器：80%→report-only、pause-all→退出；early_exit：無新訊號 <5k tokens 退場 |
| 每輪子代理數上限 | L1 期＝0；畢業後建議 ≤3（鏈深上限＝1，子代理不得再生子代理） |
| 安全級(起點) | 出生 L1；**這支的常態就是 L1**——它的價值是教會專案 state 紀律，act-small-wins 是選配不是目標 |
| 退出/完成條件 | 每run早退：無新訊號→no-op 一行 Run Log；退役：Watch List 連續兩週為空→降頻或退役（三步：移出排程/清單→STATE 標 retired→post-mortem 記 Run Log） |
| 適用訊號 | 每天早上開工混亂、訊號散落多處、團隊第一次養迴圈、想建立 LOOP/STATE/PORTFOLIO 鷹架 |
| 不適用訊號 | 專案訊號週級才變化（改 LP-08 或週頻）；需要分鐘級即時反應（那是 LP-02/03 的事） |

### LP-02 PR 保姆 PR Babysitter

| 欄位 | 內容 |
| --- | --- |
| 名稱 | LP-02 PR 保姆 PR Babysitter |
| 目的 | 單一目標：盯所有 open PR 的 review 意見、CI 失敗、merge conflict，推動每個 PR 前進到可合併狀態。non-goals：不開新功能 PR、不重寫別人的 PR、不 merge（The loop only proposes; a human actually merges）。為什麼是迴圈：PR 佇列每小時都在變。 |
| 觸發/節奏 | 若執行框架支援排程器→每 5–15 分鐘；否則手動重啟清單：**每次人工開工、午休回來、下班前各跑一輪**，並誠實標註「高頻監看能力缺失，PR 停滯偵測延遲為小時級」（此處需人工補做）。與 LP-03 同節奏帶並跑時必須錯峰啟動（相位差≥半個 cadence，見 6.2） |
| 輸入(讀什麼STATE) | 六步①〜④：`docs/loops/LOOP_PORTFOLIO.md` → `LOOP-NNN.md`（§8）→ 本迴圈 `docs/loops/STATE-NNN.md`（每 PR 條目：狀態、attempt count、`acting_on:` 欄）→ 掃其他 `STATE-*.md` 的 `acting_on`（**尤其 LP-03 CI Sweeper 的**） |
| 執行(maker做什麼) | phases：1. discover（列 open PR）2. triage（哪些卡住、為什麼）3. fix——每個 fix 在隔離工作區進行（若支援 worktree→one worktree per fix, discard after REJECT；否則獨立目錄複本＋檔案鎖約定＋人工合併清單，此處需人工補做合併）4. verify 5. notify（開 draft PR／留言，不 merge） |
| 驗證(checker怎麼擋) | verifier 子代理（不同 context、預設 REJECT）五項全過才 APPROVE：Scope（只改相關檔、無 denylist、無無關編輯）／Intent／Tests（**親自跑並附輸出片段，Do not trust implementer's claim**）／No cheating（無停用測試）／Risk（中風險以上即使測試過仍建議人審）；鐵律：Never let the implementer sub-agent mark its own work done |
| 人類閘門 | `security/**`、`payments/**`、`auth/**` 路徑一律升級；同一 PR 第 3 次修復嘗試失敗→帶完整 context 升級並停手；行為變更類 fix 一律人審 |
| 預算上限與斷路器 | noop 3k / report 80k / action 250k；日上限 2M；單輪 action ≤ 日上限 60%；斷路器三件套；early_exit：**必須**（無 open PR→<3k 退場） |
| 每輪子代理數上限 | L1 期＝0；畢業後建議 ≤3（鏈深上限＝1，子代理不得再生子代理） |
| 安全級(起點) | 出生 L1（week one report-only：只報「哪些 PR 卡在哪」）；畢業起點 L2：需 1–2 週 L1 期 triage 準確率實證（假陽性 <30%），走一張 APPROVED-MMM |
| 退出/完成條件 | 每run早退：無 open PR→no-op；退役：PR 流量降到 <2/週 持續一個月→退役改用 LP-01 順帶盯 |
| 適用訊號 | PR 積壓、review 等待時間長、CI 失敗掛在 PR 上沒人理 |
| 不適用訊號 | PR 流量極低；token 預算緊（這是高頻高耗模式：5m cadence＝288×/日的線性乘數）；release week 想凍結；**CI 紅色期間——本支自動 no-op 讓位 LP-03 CI Sweeper**（錯峰規則見 6.2；對向條款見 LP-03 不適用訊號欄） |

### LP-03 CI 清道夫 CI Sweeper ⚠ 有前置條件

| 欄位 | 內容 |
| --- | --- |
| 名稱 | LP-03 CI 清道夫 CI Sweeper |

| 驗證(checker怎麼擋) | verifier 在隔離工作區重跑失敗測試＋相關測試套件並附輸出；REJECT 鐵條：Never disable tests to make CI green、不得刪改測試斷言讓其通過（sys.exit(0)／patch 測試框架類作弊有實證，checker 必查測試檔 diff） |
| 人類閘門 | infra failures、security tests、max 3 attempts、同一項 48h 內升級 2 次以上→整項凍結待人 |
| 預算上限與斷路器 | token 成本：**極高**；日上限 1M；單輪 action ≤ 日上限 60%；斷路器三件套；early_exit：**必須**（CI 綠→立即退場） |

| 輸入(讀什麼STATE) | 六步①〜④：`docs/loops/LOOP_PORTFOLIO.md` → `LOOP-NNN.md`（§8）→ 本迴圈 `docs/loops/STATE-NNN.md`（上次回顧游標）→ 掃其他 `STATE-*.md` 的 `acting_on`；另讀：各迴圈 `STATE-*.md` 尾節「## Run Log」近 30 天 ＋ CONVERSATION_LOG 近期段落 ＋ 既有 memory/skill 清單 |
| 執行(maker做什麼) | phases：1. 掃近期紀錄找重複流程與已驗證事實 2. 產 skill 草稿／memory 修訂提案（每條附出處紀錄位置）3. 列過期候刪清單 4. 全部進人審佇列 |
| 驗證(checker怎麼擋) | checker 查每條新 memory 可追溯到具體紀錄（**無出處不收**）、候刪清單逐條確認「真的沒在用」（近 30 天各 Run Log 無引用）、skill 草稿與既有 skill 無重複 |
| 人類閘門 | 所有 memory/skill 的實際寫入（記憶是意圖層資產，永遠人類批准）；候刪執行 |
| 預算上限與斷路器 | 每 run ~50k（預設值，需實測校正）；週上限 100k；單輪 action ≤ 上限 60%；early_exit：無新沉澱項→no-op |
| 每輪子代理數上限 | L1 期＝0；畢業後建議 ≤3（鏈深上限＝1，子代理不得再生子代理） |
| 安全級(起點) | 出生 L1；**常態 L1 propose-only** |
| 退出/完成條件 | 每run早退：游標後無足量新紀錄；退役：專案封存前做最後一輪總沉澱 |
| 適用訊號 | 專案跑了數週、Run Log 與對話已累積、同樣的事教過 agent 兩次以上、艦隊其他迴圈的 post-mortem 需要歸檔 |
| 不適用訊號 | 專案剛起步無紀錄可沉澱；紀錄系統缺席（無處可追溯出處） |

## 六、多迴圈組合守則（艦隊層規劃，NEVER SKIP）

### 6.1 六大艦隊原則（改寫自正典 multi-loop）

1. **禁止迴圈級聯（鐵律）**：任何迴圈不得觸發另一支迴圈；迴圈間只透過 STATE/PORTFOLIO 文件與人類閘門互動。
2. **One owner per resource**：同一資源（branch／資料資產／文件）同一小時內最多一支迴圈可改動。
3. **Separate state files**：每支迴圈各自一個 `docs/loops/STATE-NNN.md`（與 LOOP-NNN 同號）——絕不共寫、禁止 `*-state.md` 命名；艦隊級狀態（`loop-pause-all`、健康度、Human Inbox）一律住 `docs/loops/LOOP_PORTFOLIO.md`。
4. **Triage reports, action loops execute**：分診型（LP-01/07/08）只報告與**建議**，行動型（LP-02/03/04/06/09/11）才執行；LP-01 report-only——它「建議」其他迴圈的當日執行優先序，寫入 digest 與手動重啟清單，由人類決定啟動。
5. **Shared denylist**：denylist 基線（glob 語法：`.env*`、`**/secrets/**`、`**/credentials/**`、`auth/**`、`payments/**`、`**/migrations/**`…）住 `LOOP_PORTFOLIO.md` 艦隊級約束節，各 LOOP-NNN.md 引用並可加嚴（只准加嚴不准放寬），並以一句 prompt 編進每支迴圈的執行 prompt（⑦最小變更實作起草）：「Do not modify files matching the denylist. Escalate to human with context.」
6. **Aggregate token budget**：全艦隊日預算集中記帳於 `LOOP_PORTFOLIO.md` 艦隊總表，各迴圈自身預算內嵌於 `LOOP-NNN.md` §8——不是各自為政，也不另立預算檔。

### 6.2 碰撞避免與錯峰

- **`acting_on:` 協定（六步④⑤）**：每支行動型迴圈開工前，先掃所有其他 `STATE-*.md` 的 `acting_on`——若他人 `acting_on` 同一資源→本輪讓位並記 Run Log；自己動手前在本迴圈 STATE-NNN.md 寫 `acting_on: <資源ID>`。
- **隔離工作區降級鏈**：worktree（若支援）→ 獨立目錄複本＋檔案鎖約定（STATE 檔內 lock 欄）＋人工合併清單（此處需人工補做）→ 最低限：錯開節奏＋人工序列化執行。孤兒工作區必清——A loop that leaves orphaned worktrees behind is a loop you will regret。
- **錯峰（唯一允許條件）**：**同節奏帶允許多支迴圈的唯一條件＝錯峰啟動（相位差≥半個 cadence）＋各自宣告 acting_on＋CI 紅色期間 PR Babysitter 自動 no-op 讓位 CI Sweeper**。清理/同步類（LP-06/11/12）一律 off-peak。
- **共享 Human Inbox**：跨迴圈歧義項統一放 `docs/loops/LOOP_PORTFOLIO.md` 的 Human Inbox 節（含 `Loop action:` / `Human decision:` 欄），並接到紀錄系統的 HANDOFF/CONVERSATION_LOG；項目滯留 >24h → 下輪任何迴圈都要重提告警。

### 6.3 衝突優先序（資源相撞時誰讓誰；正典版本住 LOOP_PORTFOLIO.md 艦隊級約束節）

```
1. LP-03 CI 清道夫          ← Red main blocks everything（CI 紅色期間 LP-02 自動 no-op 讓位）
2. LP-02 PR 保姆            ← 同一 branch 若 LP-03 acting_on → 本輪讓位並記 Run Log
3. LP-04 依賴清掃           ← CI 紅時整支暫停
4. LP-09 資料品質           ← 上游資料先於下游內容
5. LP-06 合併後清理 / LP-05 變更日誌 / LP-10 內容產線   ← off-peak
6. LP-11 文件同步 / LP-12 記憶沉澱                      ← 最低位，off-peak，全部讓路
7. LP-01 每日分診 / LP-07 議題分診 / LP-08 研究監測     ← 只報告不搶資源，隨時可跑
   （LP-01 report-only：它「建議」其他迴圈的當日執行優先序，寫入 digest 與手動重啟清單，
     由人類決定啟動——迴圈不得觸發迴圈）
```

### 6.4 總預算分配（📋 fill-in，記入 docs/loops/LOOP_PORTFOLIO.md 艦隊總表；各迴圈自身預算內嵌 LOOP-NNN.md §8）

```markdown
| 迴圈 | Max runs/day | Max tokens/day | 每輪子代理數上限 | 子代理鏈深度上限 | 目前 L 級 |
| --- | --- | --- | --- | --- | --- |
| （每支一列）＿＿ | ＿＿ | ＿＿ | ＿＿（L1 期＝0，建議≤3） | 1（不得再生子代理） | L＿ |
| **艦隊合計** | | ＿＿（≤ 團隊日預算的 ＿%） | | | |
```

- **60% 規則**：單輪 action 預算不得超過日上限的 60%，否則該輪開始前即降為 report-only（絕不在動手中途切斷）。
- **80% 規則**：週中任一迴圈耗掉日上限 80% → 該迴圈自動降為 report-only 到隔日。
- **超額三步**（寫進每份 LOOP-NNN.md）：1. 暫停排程／從手動清單畫掉高頻項 2. 事件追記本迴圈 `STATE-NNN.md` 尾節「## Run Log」（一行 JSON）3. 開人類票（記入 `LOOP_PORTFOLIO.md` Human Inbox）。
- **Kill switch**：`docs/loops/LOOP_PORTFOLIO.md` 設 `loop-pause-all: true` 旗標——任何迴圈每輪第①步必查，為 true 即刻退出；復跑條件由人類在 PORTFOLIO 清除後生效。
- **計量降級**：若執行框架提供用量計量→以實數執行 80% 規則；否則降級為**結構性配額**：每輪處理項目數上限（建議 1–3）、工具呼叫次數上限、子代理數上限、每項目 max 3 attempts；Run Log 改記「輪數/項目數/工具呼叫數」，token 實帳標「此處需人工補做——由人類從框架帳單回填校準」。
- **理解債稽核**：每支迴圈 STATE-NNN.md frontmatter 的 `last_digest_ack:{日期/owner}` 由人類讀完 digest 手動回填（此處需人工補做）；六步第①步發現 ack 落後 Last run ≥3 輪→本輪自動降為 no-op，並在 digest 頂標紅「digest 連續未讀，迴圈已自動降速」；⑩驗收把 ack 回填紀錄納入 L1→L2 升格證據。

### 6.5 一個工程先跑哪 1–2 支（起步建議）

| 專案型態 | 第一支 | 第二支（第一支穩定 1–2 週後） | 絕不先跑 |
| --- | --- | --- | --- |
| 不確定／第一次養迴圈 | **LP-01 每日分診 @ L1**（教會 state 紀律，零 auto-merge 風險） | 視 Q1 最痛點加一支 @ L1 | 任何 5–15m 高頻模式 |
| coding 專案 | LP-01 @ L1（1d） | LP-02 PR 保姆 @ L1→L2（LP-06 可並列第三支 off-peak）；**LP-03 CI 清道夫最後加入**——等 LP-02 的 attempt limit＋verifier 兩週實證，且與 LP-02 同節奏帶必錯峰（見 6.2） | LP-03、任何 L2 出生 |
| 研究型專案 | **LP-08 研究監測 @ L1**（天生 report-only） | LP-12 記憶沉澱 @ L1（1w） | — |
| 有紀錄系統的任何專案 | LP-01 或 **LP-11 文件同步 @ L1** | LP-12 記憶沉澱 @ L1 | — |
| 資料/內容型專案 | LP-09 @ L1（先校準規則 1–2 週） | LP-10 @ L1 draft-only（**先確認人審能量**） | LP-10 無人審直接上 |

**非 coding 專案的預設首發＝LP-08 或 LP-11。**

**擴編紀律**：40% 的 multi-agent 部署死於 orchestration 複雜度而非個別 agent 能力——艦隊從 1–2 支起步，**orchestration 需求被 Run Log 證明後才擴編**；每支上線、每次 L 級升格、每次 allowlist 擴充，各走一張 APPROVED-MMM（MMM 取 `docs/REQUEST_LOG.md` 下一個流水號，與 LOOP 編號無關）。

## 七、🚫 AUTOMATIC FAIL 觸發器（⑧紅隊對任何艦隊規劃書逐條稽核）

規劃書命中任一條 → 直接打回，不進 APPROVED 流程：

1. **同一 agent／同一 context 自產自驗**（maker 兼 checker）。
2. **無 attempt cap**（缺「max 3 次嘗試→帶完整 context 升級人類」）。
3. **triage 輸出無結構**（沒有 High-Priority／Watch／Noise／State Updates 四節或等價結構）。
4. **任何迴圈未經 L1 實證直接以 L2/L3 出生**（L1 report-only week one 是鐵律）。
5. **共享 state 無 schema／多迴圈共寫同一 STATE 檔／使用 `*-state.md` 命名**（正典命名唯一：`docs/loops/STATE-NNN.md` 與 LOOP-NNN 同號）。
6. **工具連接器以全寫入權起步**（L1 期一律唯讀＋留言級，信任累積才擴權）。
7. **無 kill switch**（缺每輪第①步查 `LOOP_PORTFOLIO.md` 的 `loop-pause-all` 步驟）。
8. **用改 code 修 flake**（正解：分類→隔離/retry policy→infra 升級）。
9. **auto-merge 無 path allowlist**（allowlist 未文件化＝auto-merge 視同關閉）。
10. **無 Run Log**（STATE-NNN.md 缺「## Run Log」尾節或未每輪一行 JSON——無法回答「why did it do that Tuesday?」）。
11. **以 transcript 自述字串當完成證據**（「all tests passing」「committed 3 files」的 pattern-matching＝rubber-stamp；42% 假完成率有實證——證據必須是 checker 親跑的輸出）。
12. **一次性任務包成迴圈**（Q0 未通過）。
13. **規格單有留白欄位且未標「此處需人工補做」**（假裝完整＝欺騙）。
14. **L2+ 迴圈規格缺回滾步驟**（agent 預測自己會弄壞什麼的精確度僅 11.8%——每個 L2+ 迴圈必含 git revert 級回滾／備份還原路徑，紅隊必問「這個迴圈壞了怎麼退」；對接管線⑨系統化除錯的故障除錯協定）。
15. **approved_by 空白、或其編號在 REQUEST_LOG.md 查無對應列，卻 `status: active`**（LOOP-NNN.md frontmatter 批准欄造假）。
16. **迴圈級聯**（任何迴圈觸發另一支迴圈——迴圈間只准透過 STATE/PORTFOLIO 文件與人類閘門互動）。
