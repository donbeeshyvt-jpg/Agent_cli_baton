# 迴圈文件契約（LOOP × STATE × PORTFOLIO——文件即迴圈的家）

> **一句話**：迴圈不是 prompt，是「有記憶、有驗證、有邊界的循環程序」。這個程序需要一個家——
> 在沒有排程器、沒有 hook 的軟性環境裡，**這個家就是文件**。本契約定義三種文件的固定格式：
> `LOOP-NNN.md`（迴圈的憲法：它為什麼存在、能做什麼、不能做什麼）、
> `STATE-NNN.md`（迴圈的記憶：**對話外的持久脊椎**——模型會忘，repo 不會）、
> `LOOP_PORTFOLIO.md`（艦隊的儀表板：所有迴圈一眼看盡＋艦隊級約束的唯一母本）。
>
> **在管線中的位置**（管線全文見 `SKILL.md`）：本契約的三件套是⑪綜合編織（迴圈工程規劃書）的落地格式；
> ⑦最小變更實作起草的執行 prompt 草稿（含 denylist 一句式）貼進 LOOP §7；
> ⑨系統化除錯的故障除錯協定與回滾路徑落在 LOOP §10；⑩驗收的 Loop Ready 檢核以本契約為查核對象。
> ③規劃排程在管線中上場兩次：第一次在②任務架構後出節奏/Tier 草案，第二次在⑥方案提案後做艦隊預算加總與錯峰。
>
> **設計立場（雙引號並陳）**：
> 正方——「別再逐回合 prompt agent；設計會自己 prompt agent 的迴圈，你的工作是寫迴圈。」
> 反方——「蓋迴圈，要像打算留在工程師位子上的人，而不是只按 go 的人。」
> 迴圈是控制系統，不是自動駕駛。文件契約就是控制面板。

## 0. 檔案住址與命名（先講清楚東西放哪）

| 情境 | 位置 | 判定方式 |
| --- | --- | --- |
| 專案採用「多 agent 協作紀錄系統」 | `docs/loops/` | 指紋命中：根目錄有 `AGENTS.md` 且 `docs/00_AI_CONTEXT_INDEX.md` 存在 |
| 一般專案（無紀錄系統結構） | 專案根目錄 `loops/` | 指紋未命中即降級；**檔名、格式、規則完全相同**，只差路徑與少了 REQUEST_LOG 批准表（改用 §4.4 的降級批准法） |

命名規則（固定，機器可 grep）：

```text
docs/loops/
  LOOP_PORTFOLIO.md      ← 艦隊總表＋艦隊級約束節，全專案唯一一份；艦隊級狀態（pause-all 旗標、艦隊健康度）一律住這裡
  LOOP-001.md            ← 迴圈定義，一支迴圈一檔，NNN 三位數遞增、永不重用；預算內嵌 §8
  STATE-001.md           ← 對應的持久狀態，與 LOOP 同號；Run Log 附於其尾節「## Run Log」
  LOOP-002.md
  STATE-002.md
  ...
```

**命名唯一正典**：STATE 檔名＝`docs/loops/STATE-NNN.md`，與 `LOOP-NNN` 同號。
🚫 禁止 `*-state.md` 之類命名。docs/loops/ 下**只有三類檔**——不存在獨立的
`loop-budget.md`／`loop-run-log.md`／`loop-constraints.md`：預算內嵌 LOOP §8、
Run Log 住 STATE 尾節、艦隊級約束（denylist 基線／pause-all／衝突優先序／錯峰規則）住 LOOP_PORTFOLIO。

**🚨 硬規則（NEVER SKIP）**：

1. **一支迴圈一個 STATE 檔，絕不多迴圈共寫一檔**（Parallel Collision 防線；來源：multi-loop.md 原則 2「Separate state files」）。
2. **每輪固定六步（唯一正典；不論由排程器觸發或人工手動重啟）**：
   ①讀 `LOOP_PORTFOLIO.md`（查 `loop-pause-all` 與自身 status，非 active 立即退出）
   →②讀 `LOOP-NNN.md` 的約束與預算
   →③讀 `STATE-NNN.md`
   →④掃其他 `STATE-*.md` 的 `acting_on`（衝突→本輪讓位並記 Run Log）
   →⑤行動前寫 STATE（宣告 `acting_on`）
   →⑥收尾寫 Run Log＋人類必讀 digest。
   **跳過任一步＝該輪無效。**
   理解債剎車（附掛於第①步）：若本迴圈 STATE 的 `last_digest_ack` 落後 Last run ≥3 輪
   →本輪自動降為 no-op，並在 digest 頂標紅「digest 連續未讀，迴圈已自動降速」。
3. STATE 檔會進 git——**絕不在 STATE 檔放 credentials／token／密鑰**（safety.md：「State files are often committed — no credentials in STATE.md」）。
4. **禁止迴圈級聯（鐵律）**：任何迴圈不得觸發另一支迴圈；迴圈間只透過 STATE/PORTFOLIO 文件與人類閘門互動。
5. 三件套之外不再立檔：任何新增治理資訊，先問「它屬於哪一件」，answer 永遠是 LOOP §8／STATE 尾節／PORTFOLIO 約束節之一。

## 1. 📋 `docs/loops/LOOP-NNN.md` 模板（單一迴圈定義）

> 這是迴圈的**宣告式契約**：讀完這一檔，任何 agent／任何人都知道這支迴圈為什麼存在、多常跑、
> 誰做誰驗、碰到什麼要停、燒多少錢要斷。**填不出「目的一句話」與「非目標」的迴圈不准上線**——
> 那就是 intent debt 的定義。

★ 可直接複製的模板（`{{...}}` 與 `<...>` 為 fill-in 欄位）：

````markdown
---
loop_id: LOOP-{{NNN}}
name: {{迴圈名稱}}
pattern: {{LP-01 ~ LP-12 之一（12 模式見 patterns/迴圈模式庫.md）| custom}}
status: draft            # draft | active | paused | retired
autonomy_level: L1       # L0 僅文件化意圖 | L1 report-only | L2 assisted fixes（有 verifier）| L3 unattended-capable（有 human gates）
cadence: {{例：1d / 6h / 每次人工重啟}}
state_file: docs/loops/STATE-{{NNN}}.md
risk: {{low | medium | high}}    # HIGH_STAKES 聯集判準全文見 reference/02_角色映射與觸發.md（任一命中即 true，⑧紅隊⑩驗收不可省）
approved_by: APPROVED-{{MMM}}    # MMM 取 docs/REQUEST_LOG.md 下一個流水號，與 LOOP 編號無關；填寫前先開 REQUEST_LOG 確認該列存在。每次 L 級升格另有新批准，見 §11 批准紀錄表
owner: {{負責讀 digest 的人類姓名／代號}}
last_reviewed: {{YYYY-MM-DD}}
---

# LOOP-{{NNN}} — {{迴圈名稱}}

## 1. 目的與範圍（intent debt 防線——必填，一句都不能省）
- **Goal（一句話）**：{{這支迴圈存在的唯一目的，一句話}}
- **明確非目標（explicit non-goals）**：{{它「不做」什麼，至少 2 條；例：不做架構重構、不碰依賴升級}}
- **為什麼是迴圈而不是一次性任務**：{{持續性／週期性痛點的證據；答不出來就不該建迴圈}}
- **監看範圍（watched scope）**：{{哪些路徑／哪類訊號；glob 或明確清單}}

## 2. 模式與階段（phases）
- 對應模式：{{LP-01 ~ LP-12（見 patterns/迴圈模式庫.md）之一或 custom；custom 需在 §11 批准時附額外理由}}
- 工作階段：{{例：discover → triage → fix → verify → notify（照 registry.yaml 的 phases 欄）}}
- cron 型態：{{stateful（同一脈絡續跑、每輪 append STATE）| stateless（每輪全新、讀完即退）}}

## 3. 觸發與節奏（含降級）
- **理想節奏**：{{例：每日一次，離峰時段}}
- **若執行框架支援排程器**：{{排程設定描述，通用能力語言，例：每日 09:00 觸發本迴圈的啟動 prompt}}
- **否則（降級）**：列入 `LOOP_PORTFOLIO.md` 的「手動重啟清單」，由人類按節奏表手動啟動——
  **此處需人工補做：無排程器時，本迴圈不會自己醒來，漏跑不會有任何警報。**
- **同節奏帶共存**：若與其他迴圈同 cadence，必須符合 PORTFOLIO §3.3 錯峰規則（相位差 ≥ 半個 cadence＋各自宣告 acting_on）。
- 早退條件（early exit）：{{例：watchlist 為空 → 5k tokens 內結束本輪}}（節奏 ≤ 15m 的迴圈此欄必填）

## 4. 輸入 / 輸出契約
- **輸入**：{{每輪讀什麼：LOOP_PORTFOLIO.md ＋ STATE-{{NNN}}.md ＋ 哪些真實訊號源（CI 狀態／PR 清單／issue 清單／git log…）}}
- **輸出**：{{每輪產什麼：STATE 更新 ＋ Run Log 一筆 ＋（report｜draft PR｜升級人類）之一}}
- **人類必讀 digest（理解債防線）**：每輪輸出必附 ≤10 行摘要：改了什麼／發現什麼／什麼在等人類。
  「Traces tell you what happened, but someone still has to read them.」——沒人讀的迴圈產出視同不存在。
  人類讀完後回填 STATE-{{NNN}} frontmatter 的 `last_digest_ack`（此處需人工補做；ack 落後 ≥3 輪迴圈自動降速）。

## 5. Maker / Checker 角色指派（🚫 同一 agent 自產自驗 = AUTOMATIC FAIL）
- **Maker（實作者）**：{{指派給誰：子代理／單模型內的「最小變更實作者」人格；規則：one fix per run、
  最小 diff、Never refactor unrelated code}}
- **Checker（驗證者）**：{{指派給誰：另一個子代理（若執行框架支援生子代理；否則單模型內切換人格＋清空立場）；
  預設立場 REJECT；必須親自跑測試並附輸出片段，Do not trust implementer's claim that tests passed — run them；
  跑不了測試（環境問題）→ ESCALATE_HUMAN}}
- **完成判定分離**：判斷「done」的角色不得是寫程式的同一個角色。Verdict 枚舉：`APPROVE | REJECT | ESCALATE_HUMAN`。
- **驗證層級聲明**（照驗證層級優先序）：本迴圈的驗證屬於
  {{deterministic rules-based（最佳）| 視覺／E2E 實跑（次佳）| LLM-as-judge（官方認證 generally not robust，僅輔助，迭代上限 3–5 次）}}。

## 6. 人類閘門（Human Gates）
- 強制升級人類的條件（勾選＋補充）：
  - [ ] 觸及 security / auth / payments / billing / PII
  - [ ] 觸及 infra（Terraform / K8s production）
  - [ ] 依賴升級（供應鏈風險）
  - [ ] 單次變更 > {{N，建議 10}} 檔
  - [ ] 同一項目第 3 次嘗試仍失敗
  - [ ] {{本迴圈專屬條件}}
- 升級動作（verbatim 規則）：**「risky / ambiguous → Escalate to human with full context」**——
  升級時把完整脈絡寫進 STATE-{{NNN}} 的 High Priority 節（含 Loop action ＋ 空白的 Human decision 欄），
  並登錄到 LOOP_PORTFOLIO.md 的 Human Inbox。
- **禁止級聯（鐵律重申）**：本迴圈不得以任何形式啟動另一支迴圈；若判斷需要其他迴圈跟進
  →寫入 digest 的建議清單與 PORTFOLIO 手動重啟清單備註，由人類決定啟動。
- 通知規則：{{只在需要人類決策時通知；report-only 輪用 digest 彙整，嚴禁每輪都吵人}}

## 7. Allowlist / Denylist（glob 語法，照 safety.md）
- **Path Denylist**（母本住 LOOP_PORTFOLIO §3.4；在該基線上增列，不得刪減）：
  `.env`、`.env.*`、`**/secrets/**`、`**/credentials/**`、`**/*_key*`、`**/*_secret*`、
  `.terraform/**`、`k8s/production/**`、`**/migrations/**`、`auth/**`、`payments/**`、`billing/**`
  ＋ 本專案追加：{{...}}
- 一句式編入迴圈啟動 prompt（verbatim；管線⑦最小變更實作起草時必含）：
  **「Do not modify files matching the denylist. Escalate to human with context.」**
- **Auto-merge／自動放行 allowlist**：預設 **關閉**。若開啟需獨立 APPROVED-MMM，並列表：
  {{允許：註解／docs typo、僅測試檔的 lint auto-fix、import ordering ——
  不允許：行為變更、依賴版本 bump、lockfile、任何 denylist 路徑}}

## 8. 預算與斷路器（token debt 防線——全部填數字，不准留白；本節即預算文件，不另立 loop-budget.md）
| 欄位 | 值 | 備註 |
|---|---|---|
| tokens（no-op 輪） | {{例 5k}} | 空手而回的一輪該花多少 |
| tokens（report 輪） | {{例 50k}} | 有發現、只報告 |
| tokens（action 輪） | {{例 200k}} | 動手修＋驗證（實證校準：multi-agent ≈ 15× 單次 chat；50 步 >30×） |
| 建議日上限（daily cap） | {{例 100k}} | 原典 registry 建議值，與成本級非單調對應，以 Run Log 實帳校準 |
| 最大迭代（每項目） | 3 | 第 3 次失敗 → 帶完整 context 升級人類（硬性，不可調高） |
| 每輪處理項目數上限 | {{建議 1–3}} | 無 token 計量時的結構性配額主軸（見下方降級規則） |
| 每輪工具呼叫次數上限 | {{例 50}} | 結構性配額；Run Log 記實數 |
| 每輪子代理數上限 | L1 期＝0（升級後建議 ≤3） | 「Spawn sub-agents only when state says actionable」——僅當 STATE 標記 actionable |
| 子代理鏈深度上限 | 1 | 子代理不得再生子代理；需更深分解 → 升級人類 |
| no-progress 偵測 | {{例：連續 2 輪 Run Log 的 actions_taken=0 且 items_found 相同 → 標紅}} | 模型自估花費不可信（corr ≤ 0.39），只信 Run Log 實帳 |
- **預算通則**：單輪 action 預算不得超過日上限的 60%，否則該輪開始前即降為 report-only（絕不在動手中途切斷）。
- **斷路器三條件**（任一命中 → 本輪立即結束並記 Run Log）：①日預算耗盡 ②驗證連續 3 敗 ③碰到 denylist 或未知檔案類型。
- **80% 規則**：token 花費達日上限 80% → 本輪降為 report-only。
- **token 計量降級**：若執行框架提供用量計量 → 以實數執行 80% 規則；否則降級為結構性配額
  （上表：每輪處理項目數上限 1–3、工具呼叫次數上限、子代理數上限、每項目 max 3 attempts），
  Run Log 改記「輪數／項目數／工具呼叫數」，token 實帳標「此處需人工補做——由人類從框架帳單回填校準」。
- **kill switch**：`LOOP_PORTFOLIO.md` 出現 `loop-pause-all: true` 或本檔 status ≠ active → **立即退出，什麼都不做**（六步第①步）。

## 9. 安全級與升級條件（L1→L2→L3，證據本位）
- 出生等級：**一律 L1 report-only**（🚫 任何迴圈直接以 L2/L3 出生而無 L1 實證紀錄 = AUTOMATIC FAIL；
  「Never skip L1 for a new pattern on a production repo.」）
- L1 → L2 條件：{{例：連續 1–2 週（或等效 N 輪）triage 準確、假陽性 < 30%，證據 = Run Log ＋ 度量節}}
  ＋ owner 的 `last_digest_ack` 有持續回填紀錄（digest 有人讀是升格證據；ack 長期落後不得升格）
  ＋ 驗收實證者出具證據 ＋ 新的 APPROVED-MMM。
- L2 → L3 條件：{{例：2–4 週 L2 運行、零意外變更}} ＋ 具備 denylist ＋ §8 預算數字齊全且 Run Log 有實帳
  ＋ verifier 實跑證據 ＋ 新的 APPROVED-MMM。
- **Loop Ready 檢核**：升格批准前必跑（預設 NEEDS WORK）；門檻 L1≥38 / L2≥58 / L3≥78，
  逐項權重見 reference/02_角色映射與觸發.md。
- 降速／暫停／處決判準（照 operating-loops.md）：
  減速＝週中預算已破 80%、假陽性 > 30%、同項 48h 內升級 2+ 次、release week；
  暫停＝production incident 進行中、schema migration、關鍵 reviewer 不在；
  處決＝連續 S2 失敗、cost > value 兩週、團隊已靜音所有通知。

## 10. 回滾與終止（L2 以上必填——「這個迴圈壞了怎麼退」是紅隊必答題；管線⑨系統化除錯的落地節）
- **故障除錯協定**：{{每輪失敗時的固定除錯順序：讀 Run Log 最近 3 筆 → 比對 STATE 與真實 repo →
  重現最小案例 → 修復或升級人類；除錯輪也要照六步協定走}}
- **回滾路徑**（git revert 級）：{{例：所有變更走 draft PR，revert = 關 PR；已 commit 則 git revert <range>；
  假設每輪都可能有無聲回歸——agent 預測自己會弄壞什麼的精確度僅 11.8%}}
- **終止（kill）程序三步**：①移除排程／從手動重啟清單劃掉 ②STATE-{{NNN}} 標 `status: retired` 並歸檔
  ③post-mortem 寫回 {{docs/DEV_LOG.md 或 docs/HANDOFF.md 的 Known Risks；無紀錄系統則寫進本檔尾註}}。
````

## 4. 與「多 agent 協作紀錄系統」的整合規約

> 原則：**迴圈文件「住進」紀錄系統，但不覆寫、不依賴它**（立場：配合、不覆寫、不依賴）。
> 紀錄系統的 pre-commit / check-consistency 若已安裝，迴圈的 commit 一樣被擋——那是它的 enforcement，不是本契約的依賴。

### 4.1 住址
指紋命中（根目錄 `AGENTS.md` ＋ `docs/00_AI_CONTEXT_INDEX.md`）→ 三件套住 `docs/loops/`。未命中 → 專案根 `loops/`，格式不變。

### 4.2 批准制（APPROVED-MMM——迴圈世界的 human gate 與紀錄系統天然同構）
MMM 取 `docs/REQUEST_LOG.md` 下一個流水號，**與 LOOP 編號無關**；填寫前先開 REQUEST_LOG 確認該列存在。
以下事件**每件各拿一筆** APPROVED-MMM（沿用其正典表格列，row 是 source of truth）：

| 事件 | REQUEST_LOG 的「Feature scope approved」欄寫法 | 附帶證據 |
|---|---|---|
| 迴圈上線（一律 L1） | `docs/loops/LOOP-NNN 上線 @ L1 report-only` | LOOP 檔通過 §1 的 AUTOMATIC FAIL 自檢 |
| L1→L2 升格 | `LOOP-NNN 升格 L2（assisted fixes）` | STATE 度量節（含 last_digest_ack 紀錄）＋ 驗收實證者裁決（Loop Ready 門檻 L2≥58，權重見 reference/02_角色映射與觸發.md） |
| L2→L3 升格 | `LOOP-NNN 升格 L3（unattended-capable）` | 2–4 週零意外變更的 Run Log 實帳（Loop Ready 門檻 L3≥78） |

| 日預算調升 | `LOOP-NNN daily cap {{舊}}→{{新}}` | 近期實耗數據 |

### 4.3 回寫（迴圈的重大發現流回紀錄系統，防止兩套記憶分岔）
- **升級人類的項目** → 寫 STATE 的 High Priority ＋ PORTFOLIO 的 Human Inbox，**並**在 `docs/HANDOFF.md` 的

- **明示**：常規輪的 digest 只住 STATE 與 digest 本身，**不回寫**紀錄系統；僅上列事件類型回寫。
- **重大發現／post-mortem／迴圈處決** → 追加到 `docs/DEV_LOG.md`（若存在；否則 `docs/HANDOFF.md`），一行事件＋指回 STATE-NNN。
- **影響「下一個安全任務」的發現** → 更新 HANDOFF 的 `Next Safe Task`。

```text
   （approved_by 的編號在 docs/REQUEST_LOG.md 有對應列？查無即 🔴）

   且 digest 頂有標紅「digest 連續未讀，迴圈已自動降速」——未降速＝理解債剎車失靈，雙重 🔴。

   （State Rot——「loop acts on ghosts」；能跑 shell 就實際查，不能就標 ⚪）
10. 有無兩支迴圈的 acting_on 指向同一標的？（Parallel Collision → 立即 🔴；
    同節奏帶的迴圈有無滿足錯峰條件：相位差≥半個 cadence？）
11. Run Log 近 7 天 tokens（或降級時的 輪數/項目數/工具呼叫數）加總 vs LOOP §8 日預算與總表
    aggregate cap：有無超支未記異常？有無單輪 action 超過日上限 60% 卻未降 report-only？
【契約自身】
12. 各 LOOP §7 的 denylist 是否仍為總表 §3.4 母本的超集（只增未刪）？
13. Run Log 的 outcome 是否都在四值枚舉內、30 天修剪有無執行？
14. 抽查最近 3 輪：每輪是否完整走過六步協定？（①有讀 PORTFOLIO——pause-all 期間有無迴圈照跑；
    ④衝突時 Run Log 有無讓位紀錄；⑤acting_on 是否行動前宣告；⑥每輪有無 Run Log＋digest）
    任一缺 → 🔴（該輪按契約視為無效輪）。
```
