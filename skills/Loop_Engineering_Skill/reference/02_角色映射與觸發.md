# 角色映射與觸發｜NEEDS_LOOP 旗標契約 × Router 增列 × 12 角色迴圈工程管線

> 本檔是 **Loop_Engineering_Skill 與 Agent_OS_Skill 的接口件**（本檔路徑：`Loop_Engineering_Skill/reference/02_角色映射與觸發.md`）。它回答四件事：意圖解碼者**怎麼判斷**「這題該用迴圈工程」（`NEEDS_LOOP` 旗標）、Router 真值表**加哪一列**、12 角色在「迴圈艦隊規劃」中**各自接什麼棒**、以及最終交付「迴圈工程規劃書」長什麼樣。
>
> 全程**軟性 prompt 驅動**：不安裝、不依賴任何 hook／排程器；若執行框架支援排程器／子代理／隔離工作區／token 計量則對應使用，否則照 §A-6 降級鏈誠實降級，缺什麼標「**此處需人工補做**」，絕不假裝。
>
> 世界觀雙引並陳，缺一不可：
> - 正方：「別再逐回合 prompt agent；設計會自己 prompt agent 的迴圈，你的工作是寫迴圈。」
> - 反方：「蓋迴圈，要像打算留在工程師位子上的人，而不是只按 go 的人。」
>
> **迴圈是控制系統，不是自動駕駛。** 本管線規劃的是「一個工程的迴圈艦隊」，不是單一迴圈。
>
> 治理文件三件套（唯一正典，契約全文見 `Loop_Engineering_Skill/reference/01_迴圈文件契約.md`）：`docs/loops/` 下**只有三類檔**——`LOOP-NNN.md`（單迴圈定義，預算內嵌 §8）、`STATE-NNN.md`（持久狀態，與 LOOP-NNN 同號；Run Log 附於其尾節「## Run Log」每輪一行 JSON）、`LOOP_PORTFOLIO.md`（艦隊總表＋艦隊級約束節[denylist 基線／pause-all／衝突優先序／錯峰規則]＋健康度）。**不存在**獨立的 loop-budget.md／loop-run-log.md／loop-constraints.md 檔；**禁止** `*-state.md` 命名。

## A. `NEEDS_LOOP` 旗標定義（意圖解碼者專用；本節為掛旗判準唯一正典）

### A-1. 旗標契約增列（★ 可直接併入 `02_Router真值表.md` A 節的 JSON schema）

```json
"flags": {
  "NEEDS_PLAN":     false,
  "NEEDS_SCHEDULE": false,   // 要時程、提醒、里程碑（時間到了由人動手）
  "NEEDS_EVIDENCE": false,
  "NEEDS_BUILD":    false,
  "NEEDS_DEBUG":    false,
  "NEEDS_LOOP":     false,   // 定期、自己跑下去：任務會重複發生／單次做完沒意義／要跨 session 持續＋驗證＋推進
                             // ——三特徵至少命中兩項才掛（判準見 A-3；語彙訊號只是線索，見 A-2）
  "HIGH_STAKES":    false    // 聯集定義見 A-1b（唯一正典全文，其他檔一律引用）
}
```

> 小模型友善寫法：在說明書最後寫「旗標：NEEDS_LOOP=true, NEEDS_EVIDENCE=true，其餘 false」即可。

2. 「定期」「自己跑下去」等語彙從 `NEEDS_SCHEDULE` 的觸發詞**移至** `NEEDS_LOOP`。
3. 真值表 B 節新增 `NEEDS_LOOP` 一列（貼上內容見本檔 §B-1）。

### A-1b. `HIGH_STAKES` 聯集（唯一正典全文——全套文件只寫這一次，其他檔一律引用本節）

> **候選 pattern 的 risk≥medium、或 token_cost≥high、或設計含 L2/L3 推進目標——任一命中即 HIGH_STAKES=true（⑧⑩ 不可省）。**

### A-2. 觸發訊號（語彙只是線索，命中後**必須**進 A-3 特徵判準，不得憑語彙直接掛旗）

### A-3. 三特徵判準（唯一正典；訊號詞只是線索，特徵才是判據；**至少命中兩項**才掛 `NEEDS_LOOP=true`）

- **(a) 任務會重複發生**：同型工作按節奏回來，不是做完就結案。
- **(b) 單次做完沒意義**：做一次只消化當下 backlog，明天又長回來（掃一次 issue ≠ 讓 issue 佇列長期乾淨）。
- **(c) 要跨 session 持續＋驗證＋推進**：需要**對話之外的持久 STATE** 才能記得「上次做到哪、試過什麼、什麼在等人類」，且每輪要自己驗證、自己推進。

### A-4. 反向出口（anti intent-debt，NEVER SKIP）

掛 `NEEDS_LOOP` 的**同時**，意圖解碼者必答一句：「**這其實一次做完就好嗎？**」（即「為什麼不用一次性任務就好？」）
- 答不出迴圈化的理由 → 不掛旗，當一般任務走（迴圈的治理成本遠高於單次任務；40% 多代理失敗案例敗在 orchestration 複雜度，不是 agent 能力）。
- 答得出來 → 這句理由**原文**寫進問題說明書 `handoff_note`，最終落進每份 LOOP-NNN.md 的 Goal／迴圈理由欄。迴圈理由未記錄＝intent debt，三種債之首。

### A-5. 與 `NEEDS_SCHEDULE` 的分界

一句話：**SCHEDULE＝「一個計畫的時程提醒」；LOOP＝「會自己重複執行＋驗證＋推進的控制系統」。**

判別句（唯一正典）：「**時間到了，是【人】動手（SCHEDULE）還是【迴圈自己】執行+驗證+推進（LOOP）？**」

| 判別維度 | `NEEDS_SCHEDULE` | `NEEDS_LOOP` |
| --- | --- | --- |
| 時間到了誰動手 | 人（或只是一次性提醒） | 迴圈自己跑：觸發→triage→行動→驗證→寫 STATE→（升級人類） |
| 交付產物 | 時程表／里程碑／提醒清單（靜態） | 控制系統：LOOP 契約＋STATE＋maker/checker＋human gate＋預算斷路器 |
| 結束方式 | 計畫做完就結束 | 沒有「做完」，只有 retire（走 kill checklist） |
| 記憶需求 | 不需要跨 session 狀態 | 必須有對話外持久 STATE（「the model forgets. The repo does not.」） |
| 例子 | 「幫我排三週開發時程」「週五提醒我交報告」 | 「每天早上自動分診 issue」「PR 積壓自動照看」「夜間無人值守跑修復」 |

**兩旗可同時亮**：迴圈需要觸發源，所以 `NEEDS_LOOP=true` 時 ③ 規劃排程者**必**上場（做 cadence／Tier 選擇／降級鏈）。反向不成立——`NEEDS_SCHEDULE=true` 不自動代表 `NEEDS_LOOP`。
**灰帶裁決（唯一正典）**：「每週提醒我 review 依賴更新」→ 提醒是 SCHEDULE；「系統自己掃依賴、自己開 patch 給我批」才是 LOOP。灰帶時**預設只掛 `NEEDS_SCHEDULE`**，把「要不要升級成迴圈」列入 `clarifications_needed` 回問使用者。

### A-6. 能力探測與降級鏈（掛旗後 ③ 必跑）

| 需要的通用能力 | 若執行框架支援 | 否則降級為 |
| --- | --- | --- |
| 排程器 | 定時觸發迴圈 | **手動重啟清單**（Manual Restart Checklist：人按清單定期手動開跑）＋標「此處需人工補做」 |
| 生子代理 | maker/checker 各自獨立乾淨 context（每輪子代理數上限：L1 期＝0，建議≤3；鏈深度上限＝1，子代理不得再生子代理，需更深分解→升級人類） | 單一模型內切換人格＋打分隔標記＋清空上一角色立場（切換衛生） |
| 隔離工作區（worktree） | 每 fix 一個 worktree，REJECT 即丟棄 | 目錄隔離＋檔案鎖約定＋人工合併清單＋標「此處需人工補做」 |
| **token 用量計量** | 以實數執行 80% 規則（花費達日上限 80% → 轉 report-only） | 降級為**結構性配額**：每輪處理項目數上限（建議 1–3）＋工具呼叫次數上限＋子代理數上限＋每項目 max 3 attempts；Run Log 改記「輪數／項目數／工具呼叫數」，token 實帳標「**此處需人工補做——由人類從框架帳單回填校準**」 |
| LOOP↔STATE 漂移偵測 | 工具比對宣告與實況 | ⑧ 的固定 prompt 檢查步驟：逐份比對 `docs/loops/` 的 LOOP-NNN.md 宣告與 STATE-NNN.md 實況，列漂移項 |

### B-2. 排序不變式（`NEEDS_LOOP` 亮起時的管線骨幹——唯一正典順序）

沿用 Agent_OS 原骨幹人格與 handoff 契約，順序照下（**③規劃排程者上場兩次：第一次出節奏／Tier 草案；第二次做艦隊預算加總與錯峰。**——全套文件一律用這句表述）：

```
① 意圖解碼者（分診＋掛旗：A-2 訊號 → A-3 三特徵 → A-4 反向出口）
 → ② 任務架構師（把工程 MECE 成迴圈候選＋依賴圖＋碰撞面）
 → ③ 規劃排程者【第一次上場】（節奏／Tier 草案；缺排程器 → 手動重啟清單）
 →〔④ 研究總管 ⇉ ⑤ 領域透鏡 ×N〕（此工程迴圈化的證據＋反例）
 → ⑥ 方案提案者（每候選的模式選型＋取捨表）
 → ③ 規劃排程者【第二次上場】（艦隊預算加總與錯峰）
 → ⑦ 最小變更實作者（起草每迴圈的執行 prompt 草稿，含 denylist 一句式）
 → ⑨ 系統化除錯者（每迴圈的故障除錯協定＋回滾路徑，如 git revert 級）
 → ⑧ 紅隊驗證者（攻擊設計：token 債／碰撞／驗證缺口／假完成）
 → ⑩ 驗收實證者（Loop Ready 檢核，預設 NEEDS WORK）
 → ⑪ 綜合編織者（迴圈工程規劃書）
 → ⑫ 記憶技能策展人（沉澱可複用模板）
```

## C. 12 角色 → 迴圈工程職責映射表（照 §B-2 唯一正典順序）

> 同一批人格、同一套 handoff 契約（summary＋metadata），只是工作對象從「一題」換成「一個工程的迴圈艦隊」。分流鐵律不變：查「主張」找 ⑧；修「失效迴圈」找 ⑨；驗「迴圈設計是否 Ready」找 ⑩。

| 角色 | 在迴圈工程規劃中做什麼 | handoff 輸出 |
| --- | --- | --- |
| ① 意圖解碼者 | **分診哪些工作值得迴圈化＋掛旗**：跑 A-2 訊號詞＋A-3 三特徵（至少命中兩項）＋「What hurts right now?」決策樹，逐痛點判「迴圈化／一次性任務／不做」，並強制回答反向出口「這其實一次做完就好嗎？」（A-4）；灰帶預設只掛 `NEEDS_SCHEDULE` 並列入 `clarifications_needed` | 問題說明書＋`NEEDS_LOOP=true`＋逐痛點分診表＋逐迴圈理由（anti intent-debt，原文進 LOOP-NNN.md） |
| ② 任務架構師 | **把工程 MECE 切成迴圈候選＋依賴**：以「同節奏帶未錯峰即碰撞」為拆分警戒，標每候選的專屬 STATE 檔（`docs/loops/STATE-NNN.md`，與 LOOP-NNN 同號）、branch 擁有權、與其他候選的碰撞面（誰先誰後、誰紅了誰暫停） | 迴圈候選表（id／Goal 一句話／痛點來源／依賴／碰撞面／STATE 檔名）＋衝突優先序草案 |
| ③ 規劃排程者 | **上場兩次**（③規劃排程者上場兩次：第一次出節奏／Tier 草案；第二次做艦隊預算加總與錯峰）。**第一次（②之後）**：每候選定 cadence（cadence 是線性成本乘數：5m vs 1d＝288 倍）、stateful/stateless 型態、Tier 選擇（本機終端／CI／雲端 runtime／輕量 cron）；缺排程器→手動重啟清單；缺 token 計量→結構性配額（A-6）。**第二次（⑥之後）**：把選定方案的每迴圈預算加總對照艦隊日上限、套錯峰規則——同節奏帶允許多支迴圈的唯一條件＝錯峰啟動（相位差≥半個 cadence）＋各自宣告 acting_on＋CI 紅色期間 LP-02 PR Babysitter 自動 no-op 讓位 LP-03 CI Sweeper——並排上線順序 | 第一次：每迴圈 cadence＋Tier＋降級標註草案。第二次：艦隊總預算表（日上限加總）＋錯峰相位表＋上線時間線 |

  STATE frontmatter `last_digest_ack:{日期/owner}` 由人類讀完 digest 手動回填（此處需人工補做）；
  六步第①步發現 ack 落後 Last run ≥3 輪 → 本輪自動降為 no-op，
  並在 digest 頂標紅「digest 連續未讀，迴圈已自動降速」；
  非平凡變更強制人審；「工程師停止有意見」列為 ⑧ 常設檢查項。
- **token debt**：成本三情境＋日上限＋60% 單輪上限＋early_exit＋斷路器（§5）；
  模型自估花費不可信（corr ≤ 0.39），一律以 Run Log 實帳校準。
- 期望值管理基線：~90% 自動化＋~10% 人工清理，寫死在此、不過度承諾。

## E. 🚫 AUTOMATIC FAIL 觸發器（迴圈工程版）

> 任一命中 → ⑩ 直接判 **NEEDS WORK**、⑪ 不得交付規劃書、不得送 APPROVED-MMM。核心七條逐字檢查，擴充清單設計審查時全掃。

### 核心七條（NEVER SKIP）

1. **無人類閘門的 L2/L3 設計**：任何具寫入／行動能力的迴圈缺明文 human gates 或 denylist——「safe/allowlisted → 放行；risky/ambiguous → Escalate to human with full context」二選一必居其一。
2. **無預算斷路器**：缺成本三情境（no-op／report／action）估算、缺日上限、缺 60% 單輪 action 上限、或缺三條件斷路器（預算耗盡／連 3 敗／未知檔案）任一。
3. **無退出條件**：迴圈沒有 early_exit、終止條件或 kill 程序——「會自己跑下去」不等於「停不下來」。
4. **STATE 無時間戳**：STATE-NNN.md schema 缺 `Last run`——state rot 之源，「loop acts on ghosts」。
5. **一次規劃就上 L3**：任何迴圈未經 L1 report-only 實證紀錄就以 L2/L3 出生；同罪：升格申請不附 ⑩ 出具的可重現證據。
6. **驗證者與實作者同一人格且無切換衛生**：maker 與 checker 同 session／同指令；或單模型切換時未清空上一角色立場、未打分隔標記——「the worker does not grade its own homework」。
7. **批准號失效卻活著**：LOOP frontmatter `approved_by` 空白、或其編號在 `docs/REQUEST_LOG.md` 查無對應列，卻 `status: active`。
